## 一、背景与问题

### 1.1 当前性能现状

| 环节 | 典型耗时 | 说明 |
|------|----------|------|
| 守护进程冷启动 + 模型加载 | 30–90 s | 仅首次或切换 CPU/GPU 时 |
| 守护进程已就绪后单次合成 | 2–15 s | 与句长、GPU/CPU 有关 |
| 整句实时合成 | 随字数近似线性增长 | MOSS 非流式，长句更慢 |

告警轮询间隔常见为 **30 s**。若单次合成超过 10 s，会出现：

- 下一条告警已到来，上一条尚未播完（队列堆积）
- 运维体感「反应慢、像卡住」

### 1.2 告警文本的结构特点（以端口表为例）

标准播报句式（经 `alarm_processor` 转换后）：

```
B05柜，NE20E杠1，端口，联通
```

可拆为 **固定模板 + 可变槽位**：

| 片段 | 类型 | 是否可预热 |
|------|------|------------|
| `B05柜` | 机柜名 | 是（有限集合，历史高频） |
| `NE20E杠1` | 设备名 | 是（有限集合） |
| `端口` | 固定词 | 是（全库共用 1 条缓存） |
| `联通` / `中断` | 状态词 | 是（枚举值，仅 2–3 种） |

**结论：该场景非常适合「片段缓存 + 拼合播放」，方案有意义。**

## 二、方案是否有意义？（结论先行）

### 2.1 有意义的部分

1. **模型常驻（待机）**  
   项目已有 `moss_daemon.py`，模型加载一次、后续 HTTP 合成。  
   进一步可做：**启动应用后可选「后台预热」**，轮询开始前模型已就绪。  
   → **有意义，投入小、收益确定。**

2. **固定词 / 状态词缓存**  
   `端口`、`联通`、`中断`、`告警`、`恢复` 等词全库复用。  
   预热后合成 **0 次**，播放时直接读 WAV。  
   → **有意义，命中率接近 100%。**

3. **高频设备名片段缓存**  
   从历史告警统计 `instance_name` 出现次数，Top N（如 200）预合成。  
   新告警若命中「B05柜」「NE20E杠1」，只合成未缓存部分或零合成。  
   → **有意义，命中率与机房规模相关；典型 IDC 场景可达 60–90%。**

4. **整句模板槽位拼合**  
   句式固定为：`{机柜}，{设备}，端口，{状态}`  
   播放时按顺序拼接 4 段 PCM/WAV，中间插入 **50–150 ms 静音** 即可。  
   → **有意义，播放延迟可降至 < 500 ms（仅解码+拼接）。**

### 2.2 需谨慎的部分

| 想法 | 风险 | 建议 |
|------|------|------|
| 逐字合成 | 中文按字切分自然度差；逗号停顿难统一；缓存条目爆炸 | **不推荐逐字**；按 **词/短语** 切分 |
| 任意长句通用拆词 | 无固定模板时切分与韵律不可控 | 仅对 **已定义规则**（端口表/通用模板）启用拼合 |
| 无限缓存 | 磁盘与索引膨胀 | 设 **容量上限 + LRU**；仅保留高频与固定词 |
| 拼合无停顿 | 听起来像机器连读 | 片段间加 **短静音** 或 **交叉淡入淡出** |

### 2.3 预期收益（量级估算）

假设整句合成 8 s，拼合播放 4 段（均已缓存）：

| 模式 | 首次 | 命中缓存后 |
|------|------|------------|
| 整句实时 MOSS | ~8 s | ~8 s |
| 片段全命中拼合 | ~8 s（预热时付过） | **< 0.5 s** |
| 部分命中（仅状态未缓存） | — | ~2 s（只合成 1 个词） |

**对 30 s 轮询间隔的告警播报，该方案能显著降低「播不完」概率。**

## 四、模型待机（常驻）策略

### 4.1 与现有能力的关系

- **已实现**：`moss_daemon.py` 守护进程，模型常驻内存。  
- **待增强**：
  - 应用启动时 **可选开关**：「MOSS 启动预热」
  - 预热在 **后台线程** 完成，不阻塞 UI
  - UI 显示：`模型状态：就绪 / 加载中 / 未启动`

### 4.2 内存占用说明（给用户预期）

| 模式 | 显存/内存（约） | 响应 |
|------|-----------------|------|
| CPU 守护进程 | 1–2 GB RAM | 合成较慢 |
| GPU 守护进程 | 2–4 GB VRAM + 部分 RAM | 合成快 |

**长期待机 = 用内存换延迟**，对 4060 Ti 桌面运维场景通常可接受。

### 4.2 内存占用说明（给用户预期）

| 模式 | 显存/内存（约） | 响应 |
|------|-----------------|------|
| CPU 守护进程 | 1–2 GB RAM | 合成较慢（见 §4.3） |
| GPU 守护进程 | 2–4 GB VRAM + 部分 RAM | 合成快 |

**长期待机 = 用内存换延迟**，对 4060 Ti 桌面运维场景通常可接受。

### 4.3 纯 CPU 常驻：到底快多少？（易混淆点）

> **结论先说**：模型常驻内存，**主要省的是「每次加载模型」的时间**，**不是**把「一句合成 8 s」变成「1 s」。  
> 推理本身仍要跑完整套 ONNX，CPU 上整句仍约 **5–15 s**（端口表短句常见 **6–10 s**）。

#### 4.3.1 常驻解决什么

| 环节 | 旧方案（每次 CLI 子进程） | 守护进程已就绪 | 节省 |
|------|---------------------------|----------------|------|
| ONNX 模型加载 | 每次 **30–90 s** | **0**（启动时付一次） | **≈ 30–90 s/次** |
| HTTP/进程启动开销 | 每次 **0.5–2 s** | **≈ 0** | **≈ 0.5–2 s/次** |
| **整句推理本身** | **5–15 s** | **5–15 s** | **≈ 0**（同硬件同线程） |

因此：

- **第 2 条及以后的告警**（守护进程未重启）：常驻 vs 每次 CLI，约快 **30–92 s** —— 这是守护进程的**最大单项收益**。  
- **守护进程已热时，第 1 句 vs 第 100 句**：合成耗时**几乎一样**，不会因为「在内存里待久了」再快一截。  
- **应用启动后第一条试听**：仍要等守护进程 **首次加载 30–90 s**（除非做启动预热，把这段时间挪到后台）。

#### 4.3.2 CPU 多核（`cpu_threads`）在常驻之上还能省多少

在**模型已加载**前提下，`留 1 核` vs `单线程` 只影响**单次推理**，典型约 **20–40%**：

| 端口表短句（约 20 字） | 单线程 / 默认偏低 | 留 1 核（如 8 核机用 7 线程） | 约节省 |
|------------------------|-------------------|-------------------------------|--------|
| 整句 MOSS 合成 | ~10–12 s | ~6–8 s | **2–4 s** |

**不能**靠把线程调到 16 实现「4 个片段同时合成」——请求仍串行（见 §五-C）。

### 4.4 全方案耗时对比（相对「旧方案」）

**旧方案基准**（本项目的最初形态，便于对比）：

```
每次播报 → 拉起 moss-tts-nano CLI 子进程
         → 重新加载 ONNX 模型（30–90 s）
         → 整句合成（CPU 5–15 s）
         → 播放
```

**测试句式**：`B05柜，NE20E杠1，端口，联通`（端口表标准句，约 20 字）  
以下为**量级估算**（CPU 8 逻辑核、MOSS Junhao；GPU 为 RTX 4060 Ti 量级），实际以本机试听与 `moss_daemon` 返回的 `elapsed_ms` 为准。

#### 4.4.1 从事件到「开始出声」——按场景

| 场景 | 旧方案（CLI 每次加载） | 当前已实现（守护+整句） | 目标方案（缓存拼合等） | 相对旧方案节省 |
|------|------------------------|-------------------------|------------------------|----------------|
| **应用冷启动后第 1 条** | 35–105 s | 35–105 s（首次仍要加载守护进程） | 预热后：**0.3–0.5 s**（L1 全命中拼合） | 最多 **≈ 100 s** |
| **守护已热，首播整句** | 35–105 s | **6–10 s** | 同左或 L1：**0.3–0.5 s** | **≈ 25–95 s** |
| **守护已热，L1 片段 4/4 命中** | 35–105 s | 6–10 s（尚未实现拼合） | **0.3–0.5 s** | **≈ 35–105 s** |
| **同告警第 2–N 次（L2 整句复播）** | 35–105 s/次 | 6–10 s/次 | **≈ 0.05 s** | **≈ 35–105 s/次** |
| **GPU 守护已热，首播整句** | 35–105 s | **1–3 s** | L1 全命中仍 **0.3–0.5 s** | **≈ 32–102 s** |

#### 4.4.2 各工程优化「单项」节省（守护已热、单次播报）

假设起点为：**守护进程 + CPU 整句实时合成 ≈ 8 s**（中间态，拼合尚未上线）。

| 优化项 | 实现状态 | 典型节省（相对上一步） | 作用对象 |
|--------|----------|------------------------|----------|
| **① 守护进程常驻**（相对每次 CLI） | 已实现 | **30–90 s**（消除重复加载） | 每一次合成请求 |
| **② L1 片段缓存 + 拼合**（4/4 命中） | 待实现 | **≈ 7.5 s**（8 s → 0.5 s） | 每条命中告警的**首播** |
| **③ GPU 加速**（相对 CPU 整句） | 已实现 | **≈ 5–7 s**（8 s → 1–2 s） | 每次需调 MOSS 的合成 |
| **④ L2 整句复播缓存** | 待实现 | **≈ 0.4 s**（0.5 s → 0.05 s） | 同告警**第 2–N 次** |
| **⑤ CPU 多核调优**（留 1 核） | 已实现 | **≈ 2–4 s**（仅影响 MOSS 推理段） | 未命中缓存、需整句/片段合成时 |
| **⑥ 启动预热** | 待实现 | **不缩短合成**，把 **30–90 s** 挪到启动后台 | 仅**第一条**告警体感 |
| **⑦ 首播流式 + 缓冲** | 待实现 | **首声提前 ≈ 3–6 s**；**总时长几乎不变** | 冷句、长句、片段未命中时 |

#### 4.4.3 哪一项「减少的时间最多」？

按**不同口径**排序：

**A. 相对最初「每次 CLI」旧方案（历史总收益）**

1. **守护进程常驻** — 去掉重复加载，**每次约 30–90 s**（单项最大、已实现）  
2. **L1 片段拼合全命中** — 在守护已热基础上再省 **≈ 7–14 s**，端到端 **≈ 35–105 s → 0.5 s**  
3. **GPU** — 未命中缓存时整句 **≈ 5–7 s**（你机子有 4060 Ti 时值得开）  
4. **L2 整句复播** — 在 L1 之上再省 **≈ 0.4 s**，但相对「每次整句 MOSS」仍省 **≈ 8 s/次**  
5. **CPU 多核** — **≈ 2–4 s**  
6. **流式首播** — 改善**首声等待**，不显著缩短整段播报总墙钟时间  

**B. 守护进程已热、日常运维（稳态首播）**

1. **L1 片段拼合（全命中）** — **约 7–14 s**（**稳态下单项最大**，待实现）  
2. **GPU vs CPU 整句** — **约 5–7 s**（已实现，可选）  
3. **CPU 多核** — **约 2–4 s**  
4. **守护进程** — 此时收益为 **0**（已常驻）  

**C. 同告警重复播（冷却 / 直至恢复）**

1. **L2 整句直读** — 相对「每次整句 MOSS」**≈ 8 s/次**；相对「L1 拼合」**≈ 0.4 s/次**  
2. 重复次数越多，L2 累计收益越大（例如 1 小时内播 5 次 ≈ 省 **40 s** MOSS 时间）

#### 4.4.4 组合示例（端口表句、CPU、热词已预热）

```
旧方案 CLI 每次加载     ████████████████████████████████████████  约 45 s
守护 CPU 整句（当前）   ████████                                  约  8 s
L1 拼合 4/4 命中        ▌                                         约 0.4 s
L2 第 2 次复播          ▏                                         约 0.05 s
```

若改用 **GPU 守护整句**（未拼合）：约 **1.5 s**，仍慢于 L1 全命中拼合。

#### 4.4.5 使用建议（按你的机器）

| 配置 | 建议 |
|------|------|
| 仅 CPU、无缓存 | 务必 **守护进程 + 留 1 核**；接受首播 **6–10 s** |
| CPU + 热词预热 + L1 | 主推路径，首播 **< 0.5 s** |
| 有 4060 Ti | 未命中片段时用 **GPU** 补合成；命中后仍走拼合/复播 |
| 重复播报策略 | 优先做 **L2 整句复播**（性价比高，见 §7.1） |
| 流式 | **不要指望**缩短总时长；只缩短「多久听到第一个字」 |

- 主进程 `alarm_env` 内直接 `import onnxruntime` 加载 MOSS（环境与依赖冲突）  
- 每次播报拉起新子进程（已用守护进程解决）

## 五、片段切分与拼合规则

### 5.1 切分粒度：词/短语，非逐字

推荐 **按语义短语** 切分，与现有 `build_port_table_speech_text` 对齐：

```
原文: B05柜，NE20E杠1，端口，联通
片段: [ "B05柜", "NE20E杠1", "端口", "联通" ]
分隔: 合成时使用中文逗号停顿 → 片段间插入 80ms 静音
```

**槽位定义表（端口表规则）**

| 槽位 ID | 示例 | 来源字段 |
|---------|------|----------|
| `cabinet` | B05柜 | `instance_name` 解析 |
| `device` | NE20E杠1 | `instance_name` 解析 |
| `keyword_port` | 端口 | 固定 |
| `state` | 联通 / 中断 | `last(2)` |

### 5.2 缓存 Key 设计

```
cache_key = hash( engine + voice + text_normalized )
```

- `engine`：moss / sapi（不同引擎不共用）
- `voice`：MOSS 音色 Junhao 等
- `text_normalized`：经 `polish_speech_for_tts` 后的片段文本

磁盘路径示例：

```
cache/moss/Junhao/端口.wav
cache/moss/Junhao/B05柜.wav
cache/moss/Junhao/联通.wav
cache/index.json        # key → path, hit_count, last_used, duration_ms
```

### 5.3 拼合算法（概念）

1. 按片段顺序加载 WAV（标准 PCM，统一采样率；MOSS 输出需归一化）  
2. 每两段之间插入 `silence_ms`（默认 80，可配置）  
3. 可选：边界 10 ms 淡入淡出，减少爆音  
4. 输出单条 WAV 送入播放队列  

**未命中片段**：仅对该片段调用 `/synthesize`，完成后写缓存再拼合。

### 5.4 整句回退

以下情况 **不走拼合，走整句实时合成**：

- 规则 = 通用，且无法匹配槽位模板  
- 用户关闭「片段拼合」开关  
- 任一片段合成失败超过重试次数  
- 缓存目录不可用  

保证 **功能正确优先于性能**。

### 5.5 文本规范化与符号读法（缓存 Key 前置条件）

> UI 见 [UI优化方案 §5.5](./UI优化方案.md)；配置存 `cache/symbol_rules.json`。

#### 5.5.1 为什么必须独立成配置

缓存 Key 使用 `text_normalized`（§5.2）。若符号读法仅硬编码在 `alarm_processor.py`：

- 用户无法扩展 `/`、`#` 等读法  
- 手动热词 `NE20E-1` 与规范化后 `NE20E杠1` 可能生成 **两条 WAV**  
- 历史分析与播报润色 **规则不一致** 会导致命中率下降  

#### 5.5.2 规范化流水线

```
原始文本 (raw)
    │
    ▼
apply_device_patterns()     # B05_NE20E-1 → B05柜，NE20E-1
    │
    ▼
apply_char_rules()          # - → 杠, / → 斜杠（可配置）
    │
    ▼
apply_word_rules()          # up→联通, down→中断
    │
    ▼
polish_whitespace()         # 去多余空格、统一逗号
    │
    ▼
text_normalized  →  cache_key / 热词表「词组」列 / MOSS 合成输入
```

#### 5.5.3 默认规则（与现网 `alarm_processor` 对齐）

| 类型 | 规则 | 示例 |
|------|------|------|
| 字符 | `-` → `杠` | `NE20E-1` → `NE20E杠1` |
| 设备 | `^([A-Za-z]\d+)_(.+)$` | `B05_NE20E-1` → `B05柜，NE20E杠1` |
| 状态词 | `up` → `联通` | `端口 up` → `端口，联通` |

> 你提出的问题非常关键：**在 CPU 上若「播放比合成快」，会出现明显卡顿（buffer underrun）**。  
> 本节说明 MOSS 上游能力、风险边界，以及与本项目「片段缓存 + 拼合」如何配合。

### B.1 MOSS 上游已具备的流式能力

MOSS-TTS-Nano ONNX 路径（`third_party/MOSS-TTS-Nano`）**支持流式**，并非只能等整句合成完：

| 能力 | 源码位置 | 说明 |
|------|----------|------|
| 流式 Codec 解码 | `ort_cpu_runtime.CodecStreamingDecodeSession` | 按 audio frame 增量解码，不必等全部 frame 生成完 |
| 单 chunk 流式合成 | `onnx_tts_runtime.synthesize_single_chunk(streaming=True)` | 生成过程中通过 `on_frame` 回调逐步解码 |
| 完整流式 API | `app_onnx.OnnxNanoTTSServiceAdapter.synthesize_stream()` | 产出 `type=audio` 事件，带 PCM 与 **`lead_seconds`** |
| Web 演示 | `app.py` → `Realtime Streaming Decode` | 用 Web Audio `schedulePcmChunk` 边收边播 |
| CLI 开关 | `infer_onnx.py --realtime-streaming-decode` | 内部走 streaming decode 路径 |

**`lead_seconds` 含义**（MOSS 已计算）：

```
lead_seconds = 已产出音频时长 − 自首帧以来的墙钟时间
```

- `lead_seconds > 0`：合成 **快于** 实时，缓冲区在积累，播放安全  
- `lead_seconds ≈ 0`：产销平衡  
- `lead_seconds < 0`：播放 **快于** 合成，即将或已经卡顿  

MOSS 用 `_resolve_stream_decode_frame_budget()` **自适应攒批**：

| lead_seconds | 每次解码 frame 数 | 意图 |
|--------------|-------------------|------|
| < 0.20 s | 1 | 合成跟不上，少解码、多攒生成 |
| 0.20–0.55 s | 2 | |
| 0.55–1.10 s | 4 | |
| ≥ 1.10 s | 8 | 缓冲区充足，可批量解码 |

### B.2 本项目现状与差距

| 项目 | 现状 |
|------|------|
| `moss_daemon.py` | 仅 `synthesize()` **整句一次性** 返回 WAV，**未暴露流式接口** |
| `moss_tts.py` | HTTP 等完整 `wav_b64`，播放端无法边合成边播 |
| 告警场景典型句长 | 端口表约 15–25 字，整句 CPU 合成 5–15 s |

**结论**：流式能力在 MOSS 仓库里有，但 **alarm_speech 尚未接入**；若直接在 CPU 上流式播放而未做缓冲，你的担心 **完全成立**。

### B.3 三种播放策略对比（针对 CPU）

```
策略 A — 整句批处理（当前）
  合成完成 → 播放
  优点：零卡顿
  缺点：首声延迟 = 全句合成时间（慢）

策略 B — 裸流式（边合成边播，无缓冲）
  首帧到达即播
  优点：首声延迟低
  缺点：CPU 上极易 underrun → 卡顿、断续  ← 不推荐单独使用

策略 C — 流式 + 抖动缓冲（推荐用于「未缓存长句」）
  合成线程 → PCM 队列 → 攒够 N ms 再开播 → 播放线程消费
  优点：平衡首声延迟与流畅度
  缺点：实现复杂；CPU 上仍可能需暂停等待
```

**告警场景推荐优先级**：

```
1. 片段缓存命中 → 磁盘 WAV 直接播（策略 A 的极速版，< 0.5 s）
2. 部分命中 → 仅对「未命中片段」批处理合成，再拼合后一次播放（仍用策略 A）
3. 长句且无缓存 → 策略 C（流式 + 缓冲），或回退策略 A
```

**不建议**对端口表短句（4 个片段）使用裸流式：片段间还要拼合，流式收益小、复杂度大。

### B.4 CPU 流式播放的同步规则（设计约束）

实现「策略 C」时建议强制遵守：

#### （1）开播门控：未就绪不播放

```
min_buffer_ms = 400   # CPU 建议 400–800 ms，可配置
max_wait_ms   = 15000 # 超时则报错或回退整句批处理

循环：
  从合成队列收 PCM
  若累计时长 < min_buffer_ms → 继续等，不启动播放器
  达到门槛 → 启动播放线程，同时合成继续往队列写
```

#### （2）播放消费不得快于队列

- 播放端用 **采样时钟** 驱动（按 `sample_rate` 消耗），不要用「尽快播完」模式  
- 队列深度低于 `low_watermark_ms`（如 150 ms）时：
  - **首选**：播放线程短暂 `sleep`，等合成追上（轻微延迟，无爆音）  
  - **禁止**：重复读同一段 PCM 或跳帧（听感更差）

#### （3）合成完成 ≠ 可立即播完

必须区分两个状态：

| 状态 | 含义 | UI 文案示例 |
|------|------|-------------|
| `synthesis_done` | MOSS 已停止产出 PCM | 「合成完成」 |
| `playback_done` | 扬声器已播完队列中全部样本 | 「播放完成」 |

**进度条应绑定 `playback_done`**，否则会出现「进度 100% 但还在卡」的错觉。

#### （4）片段拼合与流式的分工

```
告警: B05柜，NE20E杠1，端口，联通

命中缓存: [✓] [✓] [✓] [✗]
                ↓
  1. 后台批处理合成「联通」一个片段（可流式，但 1 词极短，批处理即可）
  2. 四段 WAV 在内存/磁盘拼合为一条
  3. 对「拼合后的完整 PCM」一次性播放  ← 不走流式，彻底避免拼合+流式交叉问题
```

**原则：流式用于「单段长文本」；拼合用于「多段短词」。二者不要在同一播报里混用。**

#### （5）与守护进程的配合

扩展 `moss_daemon` 时建议双端点：

| 端点 | 用途 |
|------|------|
| `POST /synthesize` | 保持现有，返回完整 WAV（拼合 / 短句） |
| `POST /synthesize/stream` | SSE 或 chunked PCM，`lead_seconds` 随 chunk 返回 |

播放层根据场景选型，**默认仍走 `/synthesize`**，流式仅作 CPU 长句降级路径。

### B.5 流式是否有意义？（结合缓存方案）

| 场景 | 流式意义 |
|------|----------|
| 端口表 + 高缓存命中 | **低**：拼合后直播即可 |
| 通用长句、无缓存、CPU | **中**：流式 + 缓冲可降低 2–5 s 首声等待 |
| GPU | **中低**：整句也够快，流式收益小于 CPU |
| 预热阶段批量合成 | **无**：离线批处理，不涉及播放 |

**与缓存方案叠加后的结论**：

- **缓存 + 拼合** 解决 80% 告警的延迟问题（主路径）  
- **流式 + 缓冲** 是未命中时的 **保险丝**，不是主路径  
- CPU 上 **绝不做无缓冲流式**，必须等 `min_buffer_ms` 或等片段合成完再播

## 五-C、MOSS CPU「多核」能力说明

> 问题：MOSS 是否支持 **同时多核 CPU** 并行？

### C.1 结论摘要

| 说法 | 是否成立 |
|------|----------|
| 单次推理能用多核加速 | **是**（`intra_op_num_threads`） |
| 同一进程内多句同时并行合成 | **基本否**（有全局执行锁） |
| 多进程各跑一个 MOSS 实例并行 | **理论可以，但不推荐**（内存×N） |
| 多片段并行预合成缩短拼合等待 | **有限**（需多实例，成本高） |

### C.2 源码依据（ONNX 路径）

`ort_cpu_runtime.py` 创建 Session 时：

```python
options.intra_op_num_threads = self.thread_count   # 默认 4，对应 --cpu-threads
options.inter_op_num_threads = 1                   # 固定为 1
```

含义：

- **`intra_op`**：单个算子（如大矩阵乘）内部拆到多核 —— **这是 MOSS CPU 加速的主要方式**  
- **`inter_op`**：图里不同算子并行 —— **被设为 1，即不做算子级并行**

因此：**一次合成任务占用多核，但仍是「一句一句算」**，不是 4 句同时算。

### C.3 请求级串行锁

`app_onnx.OnnxRequestRuntimeManager`：

```python
self._execution_lock = threading.Lock()   # 同一 manager 内合成请求串行
```

`app.py` 亦有 `_cpu_execution_lock`。  
即：即使 Web 服务收到多个并发请求，**默认也会排队**，避免多线程同时碰 ONNX Session（Session 本身也非线程安全）。

### C.4 `--cpu-threads` 调优建议（告警守护进程）

#### C.4.1 留 1 核还是全部用尽？

| 策略 | 线程数 | 适用场景 | 说明 |
|------|--------|----------|------|
| **自动（留 1 核）**（默认推荐） | `逻辑核数 − 1` | 日常桌面运维、Flet UI 同机运行 | 避免合成占满 CPU 导致界面卡顿、轮询线程饿死 |
| **全部逻辑核** | `os.cpu_count()` | 无头预热机、夜间批量缓存合成 | 合成最快，但 UI 可能明显掉帧 |
| **手动 4 / 8 …** | 固定值 | 老 CPU、虚拟机、需与别的软件共存 | 按任务管理器实测调整 |

**结论（直接回答你的疑问）：**

- **不是「多句并行占满多核」**，而是 **一句合成时，单个 ONNX 算子内部用多线程**（`intra_op_num_threads`）。  
- **推荐默认「留 1 核」**，不要常态选「全部逻辑核」——尤其本机还跑着 Flet 桌面 UI、轮询线程、系统杀毒时。  
- 若机器 **只有 4 逻辑核**，留 1 核即用 **3 线程**，通常比硬上 4 线程更稳。  
- **物理核 vs 逻辑核**：`os.cpu_count()` 返回的是 **逻辑核（含超线程）**；MOSS/ONNX 按此设线程即可，无需再乘 2。  
- **超过逻辑核数**（如 16 线程配 8 核）：收益递减，上下文切换增多，**可能更慢**。

#### C.4.2 与 GPU 模式的关系

| MOSS 加速 | CPU 并行下拉 |
|-----------|----------------|
| GPU (CUDA) | **隐藏**（显卡负责计算，无需此项） |
| 纯 CPU | **显示**（红框位置，与加速并排） |

#### C.4.3 界面与守护进程传递链

```
UI「CPU 并行」下拉
  → resolve_moss_cpu_threads(预设)   # tts_config.py
  → tts.set_moss_cpu_threads(N)    # 重启 moss 守护进程
  → moss_daemon --cpu-threads N
  → OrtCpuRuntime(thread_count=N)
  → SessionOptions.intra_op_num_threads = N
```

变更线程数或 CPU/GPU 模式后，**首次试听会重新加载模型**（与切换 GPU 相同）。

#### C.4.4 参数对照表

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `cpu_threads` | `逻辑核 − 1`（默认） | 见上表 |
| 超过逻辑核数 | 不推荐 | 争抢严重，可能更慢 |
| 低于 2 | 仅调试 | 合成明显变慢 |

当前 `moss_daemon.py` 默认 `cpu_threads=4`；UI 默认改为 **「自动（留 1 核）」**，按本机核数动态计算。

**不能指望**：把 `cpu_threads` 设为 16 就能 **同时合成 16 个片段**（请求仍串行，见 C.3）。

### C.5 若要坚持「多片段并行预合成」

可行但重：

```
方案 P1 — 多守护进程（不推荐常态）
  moss_daemon --port 18764  (片段 1)
  moss_daemon --port 18765  (片段 2)
  每进程各加载一份模型 → 内存 2×/3×
  仅适合离线预热，不适合常驻

方案 P2 — 离线预热队列（推荐）
  单 daemon 串行合成 250 个热词
  夜间或空闲时跑完即可
  运行时全命中缓存，无需并行

方案 P3 — 运行时未命中片段
  串行合成 1 个未命中词（通常 0–1 个）
  然后拼合播放
```

对端口表句式，未命中数通常为 0–1，**串行足够**。

### C.6 GPU 与 CPU 并行对比

| | CPU | GPU (CUDA) |
|--|-----|------------|
| 单次合成 | 多核 intra-op，仍慢 | 单次快 |
| 并行多请求 | 串行锁，不建议多实例 | 单 Session 仍串行；多 stream 需验证，非现成能力 |
| 告警推荐 | 缓存拼合为主 + 调高 cpu_threads | 缓存拼合为主，整句也可接受 |

## 七、与「告警重复播报策略」的配合

拼合（L1 片段）解决 **单次怎么快**；重复策略 + **整句缓存（L2）** 解决 **同一告警播多次**（§7.1）。  
**推荐组合**：首次播用拼合或流式并落盘整句；第 2–N 次 **只播整句 WAV，不再调 MOSS**。

| 策略 | 行为 | 拼合是否适用 |
|------|------|--------------|
| 仅播一次（默认） | NEW 事件播 1 次 | 是 |
| 播 N 次后停止 | 同一告警 ID 计数 | 首次拼合/流式；**复播走 L2** |
| 直至恢复 | 每轮轮询若仍活跃则再播 | 同上 |
| 冷却间隔 | 同一 ID 两次播报间隔 ≥ X 秒 | 同上 |

**直至恢复 + 30 s 轮询**：若不做冷却，每 30 s 播一次。  
建议默认加 **冷却 120 s** 或 **每告警最多 3 次/小时**。

### 7.1 可进一步优化：首播流式 + 整句复播（推荐）

> **你的想法成立**：在「同一告警会播多次」的策略下，不必每次都走 MOSS。  
> **第一次**争取尽快出声（可流式）；**重复播放**直接播已拼合好的 **整句告警 WAV**。

#### 7.1.1 三层缓存（在片段缓存之上增加「整句层」）

| 层级 | 缓存对象 | Key | 典型耗时（重复时） |
|------|----------|-----|-------------------|
| L1 片段 | `端口`、`B05柜`… | `hash(engine+voice+text)` | 拼合 ~0.3 s（仍要读多文件） |
| **L2 整句告警** | **一条完整播报 WAV** | **`alarm_id` + 文本 hash** | **~0.05 s（仅解码播放）** |
| L3 流式 | 不缓存，仅首播过程 | — | 首声 1–3 s（CPU 流式+缓冲） |

L1 解决「词都见过」；**L2 解决「同一告警第 2、3、N 次播报」** — 与重复策略强相关。

#### 7.1.2 播放决策（按告警 ID + 是否首次）

```
收到播报任务 (alarm_id, speech_text, 策略=冷却重复/直至恢复…)
        │
        ▼
┌─ utterance 缓存命中？（同 alarm_id 且 text_hash 未变）──┐
│  是 → 直接 play(整句 WAV)  【重复播路径，最快】          │
└─ 否 → 首次播路径 ─────────────────────────────────────┘
        │
        ├─ 片段全命中 → 拼合 → play → 异步写入 utterance 缓存
        │
        ├─ 片段部分命中 → 补合成未命中段 → 拼合 → play → 写入 utterance
        │
        └─ 片段不可用 / 通用长句
              → 【首播】流式 + 抖动缓冲（§五-B 策略 C）
              → 流式结束同时落盘整句 WAV → 供后续复播
```

**要点**：

- **重复播**：不再调用 `moss_daemon`，不拼合，**只读一个文件**。  
- **首次播**：在写出整句 WAV 之前，可用流式降低「干等 8 s」的体感；落盘完成后即注册 L2。  
- **文本变化**：若同 `alarm_id` 但 API 反查后话术变了（`text_hash` 不同），作废旧 utterance，走首次播路径。

#### 7.1.3 何时首播值得上流式？

| 首播场景 | 是否用流式 | 原因 |
|----------|------------|------|
| 片段 4/4 命中 | **否** | 拼合已 <0.5 s，流式收益小 |
| 片段 1–2 个未命中 | **可选** | 补合成 + 拼合约 2–6 s；若改整句流式，首声可能更早 |
| 通用长句、无槽位 | **是** | 整句 MOSS 慢，流式+缓冲改善首声 |
| 仅播一次（`once`） | **低优先级** | 无复播，落盘 utterance 收益低（仍可写 L2 备试听） |

端口表短句在 **热词预热充分** 时，首播往往 **拼合即可**，流式主要用于 **冷启动 + 长句**；  
**重复播走整句缓存** 则与句长无关，**始终推荐**。

#### 7.1.4 本地文件扩展（`cache/utterances/`）

```
cache/
├── moss/Junhao/              # L1 片段（已有）
├── utterances/moss/Junhao/   # L2 整句告警
│   ├── alarm_{id}_{text_hash}.wav
│   └── index.json            # alarm_id → path, created_at, play_count
└── index.json                # L1 片段索引
```

`index.json`（utterance 层）示例字段：

```json
{
  "alarm_88421_a3f2": {
    "alarm_id": 88421,
    "text": "B05柜，NE20E杠1，端口，中断",
    "path": "cache/utterances/moss/Junhao/alarm_88421_a3f2.wav",
    "play_count": 3,
    "first_play_mode": "stitch",
    "last_played_at": "2026-06-23T22:30:00"
  }
}
```

告警 **恢复 (RECOVERED)** 后：可删除或保留该 `alarm_id` 的 utterance（建议保留至追踪移除，LRU 统一淘汰）。

#### 7.1.5 与播报策略的配合

| 策略 | 第 1 次 | 第 2–N 次 |
|------|---------|-----------|
| 仅播一次 | L1 拼合或流式；L2 可选写 | — |
| 播 N 次后停止 | 同上 | **L2 整句直播** |
| 冷却直至恢复 | 同上 | **L2 整句直播**（冷却到期且仍活跃） |
| 每 X 分钟重复 | 同上 | **L2 整句直播** |

终端摘要示例：

```
  合成: 首次 拼合 3/4 + 流式补全 | 已缓存整句供复播
  …（120s 后）…
  合成: 复播 整句缓存命中 | 42ms
```

#### 7.1.6 风险与边界

| 风险 | 缓解 |
|------|------|
| 首播流式与拼合并行实现复杂 | Phase 分步：先 L2 复播，再首播流式 |
| 磁盘双份（片段 + 整句） | 整句约 50–200KB/条；设 `max_utterance_entries` + LRU |
| 音色/引擎切换 | utterance key 含 `engine+voice` |
| 播放中打断 | 复播被插队时仍用队列，utterance 文件只读 |

#### 7.1.7 实施优先级（相对 §八）

1. **Phase B**：L1 片段拼合  
2. **Phase B+**：L2 整句落盘 + **重复播直读**（性价比高，建议早于流式）  
3. **Phase D**：首播 `/synthesize/stream` + 边播边落盘 utterance

## 八、实施阶段建议

### Phase A — 低风险（1–2 天）

- [ ] 启动可选 MOSS 预热 + 状态展示  
- [ ] 固定词表手动预热脚本（`端口`、`联通`、`中断`）  
- [ ] 播放前查缓存，命中则直接 `play(wav)`  
- [x] UI「CPU 并行」下拉（纯 CPU 时与 MOSS 加速并排，默认留 1 核）  

### Phase B — 核心拼合（3–5 天）

- [ ] 端口表规则槽位解析器  
- [ ] WAV 拼合器（pydub 或纯 PCM）  
- [ ] `cache/index.json` 与 LRU  
- [ ] 未命中片段按需合成  

### Phase B+ — 整句复播缓存（建议 1–2 天，紧接 Phase B）

- [ ] `cache/utterances/` 与 utterance 索引  
- [ ] 首次播后落盘整句 WAV；**重复播策略命中时直读文件**（§7.1）  
- [ ] 终端区分 `首次` / `复播` 合成摘要  

### Phase C — 智能化（5–7 天）

- [ ] 历史告警词频分析与批量预热 UI  
- [ ] 缓存命中率统计（终端 / 日志）  
- [ ] 与告警重复播报策略联动  

### Phase D — 可选增强

- [ ] 讯飞 / SAPI 共用文本缓存（仅缓存文本→路径映射，引擎分目录）  
- [ ] 紧急告警插队打断当前播放  
- [ ] `moss_daemon` 流式端点 `/synthesize/stream`（CPU 长句 + 抖动缓冲）  
- [ ] **首播流式 + 播毕写入 utterance**（与 §7.1 首次路径合并）  
- [ ] 播放层区分 `synthesis_done` / `playback_done` 双状态

## 九、风险与边界

| 风险 | 缓解 |
|------|------|
| 片段衔接不自然 | 固定静音时长 + 可选整句回退 |
| 磁盘占用 | 上限 500MB；LRU；仅缓存短语 |
| MOSS 音色/引擎变更 | 缓存 key 含 voice；变更时提示清空缓存 |
| 多告警并发 | 播放队列串行；拼合缩短排队时间 |
| 恢复类播报文案不同 | RECOVERED 单独模板与缓存槽位 |
| CPU 流式 underrun | 最小缓冲 400 ms + 低水位等待；短句用拼合批播 |
| 误以为多核=多句并行 | 文档与 UI 标明：`cpu_threads` 仅加速单句 |
| 重复播仍走 MOSS | 用 L2 整句 utterance 缓存（§7.1） |

## 十、验收指标

1. **延迟**：端口表句式、缓存全命中时，从事件到出声 **< 1 s**  
2. **命中率**：运行 7 天后端口表场景 **≥ 70%** 片段命中（视机房而定）  
3. **正确性**：拼合播放与整句合成听感可接受（运维盲测通过率 ≥ 90%）  
4. **稳定性**：缓存失败时自动整句回退，不丢播报
