## 一、你的问题与结论（先说答案）

### 1.1 Windows「语音包」是不是提前录好的字词和句子？

**不完全是。** 更准确地说：

- **不是**：为每个可能出现的整句告警都预先录好 WAV（那需要无限存储，也不存在这种包）。
- **而是**：微软在系统里安装了一套 **传统 TTS 引擎 + 声学资源**（音素/双音素/单元波形或参数模型），运行时根据你输入的任意文字 **现场拼接** 出语音。

所以 SAPI 快，靠的是 **轻量算法 + 系统级优化 + 流式播放**，不是靠「整句都预录好了」。

### 1.2 能否用 MOSS 提前做出「类似语音包」的内容？

**可行，而且与现有《MOSS 语音缓存与拼合方案》是同一思路。**

做法不是替换 Windows 语音包，而是在应用层增加：

```
离线/空闲时：MOSS 合成 → 保存片段 WAV 到 cache/
播报时：      查缓存 → 命中则直接拼合播放（不调神经网络）
```

除 **历史告警统计预热** 外，完全可以对 **常见固定词、状态词、模板短语** 人工维护清单并预生成——这正是告警场景里 **投入产出比最高** 的部分。

### 1.3 能否因此接近 SAPI 的速度？

**在「片段全部命中缓存」时，可以接近 SAPI 的播放延迟（约 0.3–0.5 s 开始出声），但做不到与 SAPI 完全等价。**

| 目标 | 能否达到 | 说明 |
|------|----------|------|
| 首播延迟 < 1 s | **可以**（片段全命中 + 拼合） | 不再跑 MOSS 推理 |
| 与 SAPI 一样零准备 | **不行** | 首次仍要 MOSS 批量合成或守护进程预热 |
| 任意长句都快 | **不行** | 未命中片段仍要实时 MOSS |
| 音质保持 MOSS 水平 | **可以** | 缓存的是 MOSS 合成的 WAV |

## 二、Windows SAPI 语音包的本质（深入理解）

### 2.1 它是什么

本项目中「Windows SAPI 离线」调用链（见 [速度对比文档](./Windows_SAPI与MOSS语音速度对比.md)）：

```
文本 → SAPI.SpVoice.Speak() → 系统 TTS 引擎 → 声卡（边算边播）
```

系统内置中文语音（如 **Microsoft Huihui Desktop**）通常属于：

| 类型 | 含义 | 预置内容 |
|------|------|----------|
| **拼接式（Unit Selection）** | 从大量短语音单元里挑选、拼接 | 录制好的音节/音素片段库 + 挑选算法 |
| **参数式 / 统计式（HMM 等）** | 用数学模型生成声学参数再合成 | 声学模型 + 发音词典，体积较小 |

无论哪种，**资源在安装语音包时就已经在磁盘上**；播报时 **不再加载 1GB 神经网络**，也 **不必先写出整段 WAV 再播放**。

### 2.2 它不是什么

| 误解 | 事实 |
|------|------|
| 每个汉字一条录音 | 实际是更小粒度的单元 + 运行时组合 |
| 常见句子都预录好了 | 任意新句子（如 `NE20E杠1`）也能读，靠 G2P/分词规则 |
| 和 MOSS 一样是大模型 | 模型规模与计算量小几个数量级 |
| 应用里存了语音包文件 | 资源在 `C:\Windows\Speech` 等系统目录，对应用透明 |

### 2.3 为什么告警场景下 SAPI 特别快

1. **无冷启动**：不需要 30–90 s 加载 ONNX。  
2. **流式输出**：`Speak()` 边合成边播，首字延迟极低。  
3. **短句计算量小**：二十字左右端口表话术，CPU 毫秒～秒级。  
4. **与话术结构无关**：不必拆成「机柜 + 设备 + 端口 + 状态」也能快。

## 三、MOSS「预生成语音包」——可行模型

### 3.1 定义：应用层伪语音包

在 MOSS 方案里，「语音包」应理解为：

```
cache/
├── tier0_fixed/          # 固定词：端口.wav、联通.wav、中断.wav …
├── tier1_cabinet/        # 机柜名片段
├── tier2_device/          # 设备名片段
├── tier3_phrase/          # 人工维护的常用短语
└── tier4_sentence/        # 整句兜底（可选）
```

每条缓存 = **一段固定文本 + 固定音色（Junhao）+ MOSS 参数** 离线合成得到的 **WAV 文件** + 索引元数据。

播报时若模板可拆成若干片段且 **全部命中**，播放路径变为：

```
读 WAV → PCM 拼接（加 50–150 ms 静音）→ winsound 播放
```

**此时不再调用 `moss_daemon` 推理**，速度瓶颈从「神经网络」变成「磁盘读 + 拼接」，与 SAPI 的「查表/拼接」在 **架构角色上类似**，但 **音质仍是 MOSS**。

### 3.2 与 Windows 真语音包的对比

| 维度 | Windows SAPI 语音包 | MOSS 预生成缓存（伪语音包） |
|------|---------------------|-----------------------------|
| **资源位置** | 系统目录，安装语音包 | 项目 `cache/` 目录 |
| **生成时机** | 微软预置 | 本机 MOSS 离线批量合成 |
| **粒度** | 音素/单元级（引擎内部） | 词或短语级（应用设计） |
| **任意新文本** | 现场合成，仍较快 | 未缓存部分需 MOSS 实时合成 |
| **存储** | 几十～几百 MB 级 | 取决于缓存条数（见 §5.2） |
| **播报延迟（命中）** | ~0.1–1 s | ~0.3–0.5 s（拼合） |
| **音质** | 偏机械 | 更自然 |
| **维护** | 系统更新 | 需自己预热、淘汰、版本管理 |

### 3.3 架构对照图

```mermaid
flowchart TB
    subgraph SAPI["Windows SAPI（系统语音包）"]
        T1[告警文本] --> SAPI_Engine[系统 TTS 引擎]
        SAPI_Engine --> Units[内置音素/单元库]
        Units --> Audio1[流式音频 → 声卡]
    end

    subgraph MOSS_Live["MOSS 实时（当前默认）"]
        T2[告警文本] --> Daemon[moss_daemon]
        Daemon --> ONNX[ONNX 推理 2–15s]
        ONNX --> WAV1[WAV] --> Play1[播放]
    end

    subgraph MOSS_Cache["MOSS 预生成（目标方案）"]
        T3[告警文本] --> Planner[TtsPlaybackPlanner]
        Planner --> Lookup{片段缓存}
        Lookup -->|全命中| Stitch[拼合 WAV] --> Play2[播放 0.3–0.5s]
        Lookup -->|部分未命中| Synth[MOSS 补合成] --> Save[写入 cache] --> Stitch
    end
```

## 四、可行性验证

### 4.1 技术可行性：**通过**

| 验证项 | 结论 | 依据 |
|--------|------|------|
| MOSS 能否对固定短词稳定合成 | **能** | 同一文本 + 同一 voice 可重复合成 |
| WAV 能否拼接播放 | **能** | 标准 PCM 拼接 + 短静音，方案文档已设计 |
| 端口表模板能否拆槽 | **能** | `alarm_processor` 输出格式固定 |
| 历史统计能否生成热词表 | **能** | 方案 §六 已描述 |
| 人工维护词表能否批量合成 | **能** | 与 Tier-0 固定词同一流水线 |
| 项目是否已有播放/队列基础 | **能** | `tts_engine` 队列 + `winsound` |
| 项目是否已实现缓存模块 | **尚未** | 仅设计，见方案文档「状态：方案设计」 |

### 4.2 性能可行性：**在命中条件下通过**

引用 [MOSS 语音缓存与拼合方案](./MOSS语音缓存与拼合方案.md) §4.4 估算：

| 场景 | 整句实时 MOSS | 片段全命中拼合 |
|------|---------------|----------------|
| 守护进程已热 | 6–10 s（CPU） | **0.3–0.5 s** |
| 相对 SAPI | 慢一个数量级 | **同一量级** |

**验证方法（上线后可在本机复现）：**

1. 用 `moss_daemon` 对 `端口`、`联通`、`B05柜`、`NE20E杠1` 各合成一条 WAV。  
2. 用脚本拼接为整句，计时 `winsound.PlaySound` 触发到出声。  
3. 与 `test_sapi_tts.py` 同句对比。  

预期：**拼合 MOSS 略慢于 SAPI 数百毫秒**（多读文件、拼 PCM），但远快于实时 MOSS。

### 4.3 业务可行性（告警场景）：**通过**

端口表类告警具有：

- **封闭词表**：`联通` / `中断` / `端口` / `恢复` / `告警` …  
- **有限设备名**：单机房通常数百以内，历史 Top-N 可覆盖大部分。  
- **固定句式**：`{机柜}，{设备}，端口，{状态}`。  

因此 **「常见语料预生成 + 历史补充」** 比「指望 MOSS 像 SAPI 一样对任意句都快」更现实。

### 4.4 不可行或不建议的边界

| 想法 | 结论 | 原因 |
|------|------|------|
| 预生成所有可能整句 | **不可行** | 机柜×设备×状态组合爆炸 |
| 按单字做语音包 | **不建议** | 听感断裂、缓存条目爆炸 |
| 不做拼合、只存整句 | **部分可行** | 仅适合重复播同一告警（L2 整句层），首播仍慢 |
| 缓存后永远不调 MOSS | **不现实** | 新设备名总要首次合成 |
| 用 MOSS 缓存替换 SAPI 安装包 | **概念混淆** | 层级不同，可并存、按引擎切换 |

## 五、预生成内容从哪来（三类来源）

### 5.1 Tier-0：内置固定词（人工清单，**最优先**）

与业务强相关、全库共用、条目少、命中率接近 100%：

| 类别 | 建议预生成文本 | 条数（约） |
|------|----------------|------------|
| 端口状态 | `端口`、`联通`、`中断` | 3 |
| 事件类型 | `告警`、`恢复`、`已恢复正常` | 3 |
| 连接符语气 | `柜`、`杠`（若拆槽需要） | 2 |
| 通用 | `请注意`、`发生`、`结束` | 可选 |

**实施**：写入 `assets/tts_fixed_phrases.json`，安装后或首次启动 **一键预热固定词**（方案文档 §六 Tier-0）。

### 5.2 Tier-1 / Tier-2：历史告警驱动（**次要但覆盖长尾**）

流程（已有设计）：

1. 拉取历史 500–2000 条告警。  
2. `event_to_speech` 生成播报文本。  
3. 按模板 **拆槽** 统计词频。  
4. Top 50 机柜名、Top 200 设备名片段 → 批量 MOSS 合成。

适合：**运行一周后** 命中率上升，减少「新设备名未缓存」。

### 5.3 Tier-3：人工维护「常用语」（**你提出的补充，完全可行**）

除历史统计外，运维可主动维护例如：

```json
{
  "manual_hotwords": [
    "B01柜",
    "B05柜",
    "NE20E杠1",
    "NE20E杠2",
    "核心交换机",
    "端口表告警"
  ]
}
```

**与历史的区别**：

| 来源 | 优点 | 适用 |
|------|------|------|
| 历史统计 | 自动、贴近真实 | 设备多、告警样式杂 |
| 人工常用语 | 新机房上线前可 **先录好** | 已知核心设备、演示、演练 |

两者可 **合并去重** 后进入同一预热队列。

### 5.4 存储量级粗算

假设每条片段 WAV 约 **50–200 KB**（短词），缓存 300 条：

```
300 × 150 KB ≈ 45 MB
```

可接受。若加 1000 条整句兜底（L2），需单独评估磁盘与 LRU。

## 六、推荐实施路径（与现有方案对齐）

### 6.1 阶段划分

| 阶段 | 内容 | 速度收益 |
|------|------|----------|
| **P0（已有）** | `moss_daemon` + 界面「启动时预热模型」 | 省 30–90 s 加载，**不省单次推理** |
| **P1** | Tier-0 固定词 JSON + 批量合成脚本 + 磁盘 cache | 固定词 **零推理** |
| **P2** | `TtsPlaybackPlanner` 拆槽 + 拼合播放 | 全命中句 **≈ SAPI 延迟** |
| **P3** | 历史词频 + 人工热词 UI | 提高命中率 |
| **P4** | L2 整句缓存（同告警重复播） | 重复播 **≈ 0.05 s** |

详细模块划分见 [MOSS 语音缓存与拼合方案](./MOSS语音缓存与拼合方案.md) §三、§七。

### 6.2 播放决策（简版）

```
输入：播报字符串 + 规则 ID
  │
  ├─ 能否按规则拆成槽位？ ─否→ 整句 MOSS 实时
  │
  └─ 是 → 每槽查 cache
           ├─ 全命中 → 拼合播放（快）
           ├─ 部分命中 → 仅对未命中槽 MOSS 合成 → 写入 cache → 拼合
           └─ 同 alarm_id 第 2+ 次 → 优先 L2 整句 WAV
```

### 6.3 与「选 SAPI 还是 MOSS」的关系

| 需求 | 建议 |
|------|------|
| 极致速度、能接受机械音 | 继续用 **Windows SAPI** |
| 要 MOSS 音质、又要告警快 | **MOSS + 预生成缓存拼合** |
| 新设备多、难预热 | 短期 SAPI 或 MOSS 整句实时；长期靠历史+人工热词 |

两套引擎可并存，由界面「语音引擎」切换（已实现）。

### 8.1 ① 历史预生成 — 验证要点

- 端口表告警经 `alarm_to_speech` 后可稳定拆成 4 槽（拼合方案 §5.1）  
- 500 条历史典型提取 **50–100** 个去重片段，磁盘 **5–15 MB**，在默认 500MB 上限内  
- 与侧栏「历史 100 条」可复用 API，**不重复造轮子**  

### 8.2 ② 热词表 — 为什么用「已制作」

用户关心的是「这条热词有没有语音文件」，而非技术术语「缓存」。UI 统一：

- **已制作** = 磁盘有 WAV，播报可拼合  
- **待制作** = 在表中但尚未 MOSS 合成  

### 8.3 ③ 存储 — 防止撑满

| 规模 | 估算 |
|------|------|
| 300 条短词 | ~45 MB |
| 500 条 | ~75 MB |
| + L2 整句 200 条 | +20–40 MB |

默认上限 **500 MB / 500 条**；90% 黄警、100% 禁新预合成。详见拼合方案 §5.7。

### 8.4 ④ 手动热词 — 与历史的关系

```
系统固定词 ⊂ 热词全集
历史分析词 ⊂ 热词全集
手动添加词 ⊂ 热词全集
```

同文本 **去重 Key = normalize_for_cache(text)**；手动可 `pinned` 防 LRU。

### 8.5 ⑤ 符号读法 — 必须与缓存一致

现网 `alarm_processor` 已把 `-` 读作「杠」（`NE20E-1` → `NE20E杠1`）。  
扩展为可配置后：

- 历史分析、手动添加、播报、**cache_key** 共用 `normalize_for_cache()`  
- UI 提供 **输入 → 输出** 预览，避免用户困惑  

默认规则与现网行为 **向后兼容**。

## 十、风险与听感注意

1. **拼合缝**：片段间加 **50–150 ms 静音**，避免连读。  
2. **韵律一致**：同一 voice、同一 MOSS 参数合成。  
3. **文本规范化**：缓存 Key 必须用 `normalize_for_cache()`（§8.5）。  
4. **引擎升级**：模型/voice 变更时需 cache 版本失效重算。  
5. **符号读法变更**：可能导致旧 Key 与新热词不一致，需 UI 提示是否重制。  
6. **不等于 SAPI**：未命中时仍可能 **6–10 s** 级 MOSS 推理。
