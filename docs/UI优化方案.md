> 文档版本：2026-06-29 **v6.2**（数据终端垂直滚动 + v6.1 五大功能细化）

> 关联方案：[MOSS语音缓存与拼合方案.md](./MOSS语音缓存与拼合方案.md)、[MOSS预生成语音包可行性方案.md](./MOSS预生成语音包可行性方案.md)、[Windows_SAPI与MOSS语音速度对比.md](./Windows_SAPI与MOSS语音速度对比.md)

## 一、设计原则

### 1.1 保持不变的风格（用户已认可）

| 元素 | 规范 | 说明 |
|------|------|------|
| 深色底 | `#0f1117` | 页面背景 |
| 卡片面 | `#181c24` / `#1f2530` | 侧栏卡片、输入框底 |
| 边框 | `#2a3140` | 细线分隔 |
| 主强调 | `#5b9cf5` | 按钮、链接 |
| 成功 / 警告 | `#6ecf8a` / `#f0b45c` | 命中、就绪、告警 |
| 缓存专用色 | `#a78bfa` | MOSS 预缓存区标题色条 |
| 终端 | 三色圆点 + Consolas 绿字 | 「数据终端」辨识度 |
| 控件 | 高 38px、圆角 8px、字号 12 | 与 `flet_demo.py` 一致 |

**原则：告警盯盘以「数据终端」为主；MOSS 预缓存放在终端正下方，一眼可见、不必切 Tab。**

### 1.2 v6 设计目标

1. **终端 + 预缓存同屏**：右栏上下分屏，终端看事件，下方管 MOSS 片段库。  
2. **缓存可感知**：命中率、已缓存条数、最近拼合耗时可见。  
3. **预生成可操作**：固定词一键预热、历史分析、手动热词、试听片段。  
4. **与侧栏分工清晰**：侧栏管「引擎 / 模型常驻」；右下管「片段 WAV 库」。  
5. **非 MOSS 引擎时**：预缓存整块折叠或隐藏，不占空间。

## 二、整体布局 v6（相对 v5 的调整）

### 2.1 结构总览

```
┌──────────────────────────────────────────────────────────────────────────┐
│ 顶栏：告警视听助手 | ●状态 | TTS徽章 | [缓存 82%] | 日志图标              │
├────────────────────┬─────────────────────────────────────────────────────┤
│ 左栏 500px         │  右栏工作台（单列，上下分屏）                          │
│                    │  ┌─ 数据终端 ──────────────── [清空] [回到底部] ─┐  │
│ ┌ 连接配置 ─────┐  │  │  事件流 / 试听日志 / 合成摘要（Consolas 绿字）   │  │
│ └───────────────┘  │  │  ║ 固定可视高度；内容超出 → 右侧 **垂直滚动条** ║  │  │
│ ┌ 监控与语音 ───┐  │  └───────────────────────────────────────────────┘  │
│ │ 轮询/试听     │  │  ───────────── 分隔线 ───────────────────────────  │
│ │ 引擎/MOSS加速 │  │  ┌─ MOSS 语音预缓存 ─────────────── [折叠▲] ─────┐  │

│ │ 片段拼合开关  │  │  │  （flex 约 45%，min-height 280px，可折叠）     │  │
│ └───────────────┘  │  └───────────────────────────────────────────────┘  │
└────────────────────┴─────────────────────────────────────────────────────┘
```

> **v6.2**：预缓存面板占用下方高度后，终端可视区变矮；日志过长时依赖 §2.4 **垂直滚动**，避免内容被裁切或撑破布局。

**v5 → v6 关键变化**

| v5 方案 | v6 调整 | 原因 |
|---------|---------|------|
| 右栏多 Tab（终端 / 语音缓存 / 日志） | **终端 + 预缓存上下同屏** | 用户希望红框区域即预缓存，不切 Tab |
| 左栏第三张「语音缓存」大卡片 | **改为顶栏徽章 + 右下主面板** | 避免侧栏过长、功能重复 |
| 热词详情只在 Tab | **热词表放在终端下方** | 运维可同时看日志与缓存状态 |
| 「落盘日志」独立 Tab | **保留顶栏图标一键 tail**（可选 Phase 4 加 Tab） | 日志非高频，不占主屏 |

### 2.2 右栏高度分配

| 区域 | flex | min-height | 滚动 |
|------|------|------------|------|
| 数据终端 | 11 | 280px | **内容区垂直滚动**（§2.4） |
| MOSS 预缓存 | 9 | 280px | 热词表内部滚动 |
| 折叠预缓存后 | 终端 expand=1 | — | 终端占满；滚动仍保留 |

用户可点 **折叠▲** 把预缓存收成一条摘要条（`命中率 · 156 条 · 48/500MB`），终端全屏变高，但滚动能力不关闭。

### 2.4 数据终端 — 垂直滚动（v6.2 新增）

> **背景**：v6 上下分屏后，终端高度约为全屏时的 **55%**（min 280px）。轮询、历史 100 条、试听日志叠加后极易超出可视区，**必须**支持上下滚动查看完整内容。

#### 2.4.1 交互要求

| 项 | 规范 |
|----|------|
| 滚动方向 | **仅垂直**（上下） |
| 滚动条 | 内容超出可视高度时，**右侧显示垂直滚动条**（系统原生或 Flet 滚动条） |
| 鼠标 | 滚轮在终端区域内 → 上下滚动内容 |
| 触控板 | 双指上下滑动 |
| 键盘 | 焦点在终端内时：`↑` `↓` `PageUp` `PageDown` `Home` `End` 滚动 |
| 布局 | 终端**外框高度固定**（随 flex 分配），**禁止**整页右栏被日志撑高 |

#### 2.4.2 标题栏按钮

在现有 **[清空]**（`DELETE_SWEEP`）旁增加：

| 按钮 | 文案 / 图标 | 行为 |
|------|-------------|------|
| **回到底部** | `[回到底部]` 或 `↓` 图标 | 滚动到**最新一条**日志（见 §2.4.3） |
| 清空 | 已有 | 清空内容并重置滚动位置到顶部 |

可选：**行数徽章** `共 128 行`（仅日志极长时显示，不占主视觉）。

#### 2.4.3 自动滚动策略（推荐）

| 场景 | 行为 |
|------|------|
| 用户**未手动滚动**（或在底部 2 行内） | 新日志追加后 **自动滚到底部**，便于盯最新事件 |
| 用户**向上翻看**历史 | 暂停自动滚底，避免打断阅读 |
| 用户点 **回到底部** | 恢复自动滚底 |
| 清空终端 | 滚动位置回到顶部，显示欢迎语 |

实现上可用标志位 `terminal_user_scrolled_up: bool` 或检测 scroll offset 是否接近 max。

#### 2.4.4 线框（终端内部）

```
┌─ 数据终端 ────────────────────────────── [清空] [回到底部] ─┐
│ ● ● ●                                                      │
├────────────────────────────────────────────────────────────┤
│ [11:02:01] 轮询 · 活跃 12 条                               ║
│ [11:02:05] 新增告警 | B05_NE20E-1                          ║
│   文本: B05柜，NE20E杠1，端口，中断                          ║ ← 可视区
│   合成: 拼合 4/4 · 280ms                                   ║   （固定高度）
│ ...（更早的日志在上方，向上滚可见）                          ║
│ ...                                                          ║
└────────────────────────────────────────────────────────────┘
                                                          ↑ 垂直滚动条
```

#### 2.4.5 Flet 实现建议（Phase 2）

当前 `flet_demo.py` 使用 `TextField(multiline, read_only)`；分屏后建议二选一：

**方案 A（改动小）**：保留 `TextField`，外包一层固定高度容器并显式开启滚动：

```python
# 终端内容区：高度由父 Column 的 flex 约束，不随文本变高
terminal_body = ft.Container(
    content=console_output,   # TextField, multiline, read_only
    expand=True,
    clip_behavior=ft.ClipBehavior.HARD_EDGE,  # 防止内容溢出卡片
)
# TextField 在 Flet 中 multiline 默认可滚动；需保证外层有明确 max 高度
```

**方案 B（体验更好，Phase 4）**：改为 `ListView` + 逐条 `Text` 行，便于着色与「最新在上」：

```python
terminal_list = ft.ListView(
    expand=True,
    spacing=0,
    auto_scroll=False,        # 由 §2.4.3 策略控制 scroll_to
)
```

**验收**：窗口高度 780、预缓存面板展开时，连续追加 200 行日志 → 右侧出现滚动条，可滚到顶/底，布局无错位。

#### 2.4.6 与预缓存面板折叠的关系

| 预缓存状态 | 终端可视高度 | 滚动 |
|------------|--------------|------|
| 展开 | ~55% 右栏 | **更需要**滚动 |
| 折叠 | ~95% 右栏 | 仍保留；长历史时仍可能溢出 |

折叠预缓存**不能**替代滚动；只是减少滚动频率。

### 2.5 左栏职责（监控与语音）

侧栏保留 **操作入口**；右下保留 **状态与列表详情**。

| 区块 | 位置 | 内容 |
|------|------|------|
| 连接配置 | 左栏上 | HOST、Token |
| 轮询 / 试听 | 左栏中 | 已有 |
| 引擎与规则 | 左栏中 | 语音引擎、文本规则、MOSS 加速、CPU 并行 |
| MOSS 模型 | 左栏中 | **启动时预热模型**（守护进程常驻，已实现） |
| MOSS 播报 | 左栏中 | **启用片段拼合** 开关、静音间隔（高级折叠） |
| 试听进度 | 左栏试听下 | 合成/预热 **已用时、总耗时**（已实现） |
| 播报策略 | 左栏下 | 出现告警时、启用语音播报 |

**不在侧栏重复**：热词表格、批量预热进度详情、磁盘统计 → 全部在 **右下预缓存面板**。

## 五、五大功能 — 细化设计

### 5.1 ① 拉取历史数据 → 语音预生成

**目标**：从监控平台历史告警中自动提取高频片段，批量 MOSS 合成并写入热词表。

#### 5.1.1 入口与前置条件

| 项 | 说明 |
|----|------|
| 主按钮 | `[从历史生成热词]`（操作栏左侧，强调色） |
| 前置 | 已填 HOST/Token；建议先点侧栏「历史 100 条」验证连通 |
| 依赖 | `AlarmApiClient.fetch_history` + `alarm_to_speech` + 槽位拆分 |

#### 5.1.2 向导弹窗（三步）

**步骤 1 — 拉取范围**

```
┌─ 从历史生成热词 ─────────────────────────────────────┐
│  拉取条数  [ 500 ▼ ]  （100 / 500 / 1000 / 2000）    │
│  文本规则  [ 自动匹配 ▼ ]  （与侧栏同步）             │
│  □ 分析后自动开始预合成（否则仅导入热词表）           │
│  □ 跳过已制作条目                                    │
│                        [取消]  [下一步：分析]         │
└──────────────────────────────────────────────────────┘
```

**步骤 2 — 预览分析结果**

```
┌─ 分析完成 ───────────────────────────────────────────┐
│  共分析 500 条告警 → 提取片段 87 个（去重后）         │
│  ┌ 新增热词预览 ─────────────────────────────────┐ │
│  │ 机柜 ×12   设备 ×58   状态 ×0（已有）  固定 ×0   │ │
│  │ 示例: C12柜, NE20E杠3, B08柜 …                   │ │
│  └────────────────────────────────────────────────┘ │
│  预计新增 WAV 约 6.5 MB  │  预计合成耗时 ~15 分钟     │
│                        [上一步]  [确认并导入]        │
└──────────────────────────────────────────────────────┘
```

**步骤 3 — 后台预合成**

- 写入 `hotwords.json` → `hot_from_history`  
- 若勾选自动预合成：启动 `WarmupJob`，面板 D9 显示进度  
- 终端一行：`[缓存] 历史分析完成，新增 87 条，预合成 37/87…`

#### 5.1.3 后端流程（与拼合方案 §6 对齐）

```
fetch_history(N)
  → 每条 alarm_to_speech(rule)
  → split_into_slots(rule)        # 端口表 4 槽 / 通用 2 槽
  → normalize(polish + symbol_rules)
  → 词频统计 → Top 机柜/设备
  → merge hotwords.json
  → optional: WarmupJob.enqueue(pending)
```

#### 5.1.4 与侧栏「历史 100 条」关系

| 按钮 | 作用 |
|------|------|
| 侧栏 **历史 100 条** | 拉取并在 **数据终端** 展示，顺带写入内存缓存 |
| 预缓存 **从历史生成热词** | 专用于 **词频分析 + 热词入库 + 预合成**，终端只写摘要 |

可复用同一次拉取结果：若 5 分钟内已拉过历史，弹窗提示「使用缓存的 100 条 / 重新拉取」。

### 5.2 ② 热词表 — 直观可视化设计

**目标**：一眼看出「哪些是热词、是否已制作 WAV、占用多大、播过几次」。

#### 5.2.1 为什么用「已制作 / 待制作」而不是「已缓存」

| 旧文案 | 新文案 | 原因 |
|--------|--------|------|
| 已缓存 | **已制作** | 用户理解：MOSS 已经生成过语音文件 |
| 待合成 | **待制作** | 已在热词表但尚无 WAV |

#### 5.2.2 列表列定义

| 列 | 宽度 | 内容 | 说明 |
|----|------|------|------|
| **词组** | flex 2 | 规范化后的文本 | 如 `NE20E杠1`（已应用符号读法） |
| **来源** | 72px | 系统 / 历史 / 手动 | 图标+文字：`🛠系统` `📊历史` `✎手动` |
| **状态** | 88px | 徽章 | 见 §5.2.3 |
| **大小** | 64px | `42 KB` 或 `—` | 已制作行显示文件大小 |
| **命中** | 48px | 数字 | 播报拼合命中次数 |
| **操作** | 120px | 按钮组 | 依状态变化 |

#### 5.2.3 状态徽章（强化视觉）

| 状态 | 徽章 | 行背景（hover 前） | 可操作 |
|------|------|-------------------|--------|
| **已制作** | `● 已制作` 绿 | 无 | 试听、删除 |
| **待制作** | `○ 待制作` 灰 | 浅灰 `#1a1e28` | 合成、删除 |
| **制作中** | `⟳ 制作中` 蓝 | 浅蓝 `#1a2535` | 无（禁用） |
| **失败** | `✕ 失败` 红 | 浅红 `#2a1a1a` | 重试、删除 |

#### 5.2.4 筛选 Tab（顶栏快捷切换）

```
[ 全部 156 ] [ 已制作 144 ] [ 待制作 12 ] [ 失败 0 ] | [ 固定 ] [ 历史 ] [ 手动 ]
```

- 数字实时更新  
- 「待制作」Tab 默认排序：**命中预期高者优先**（历史词频降序）  
- 搜索框：模糊匹配 `词组` 列  

#### 5.2.5 制作进度环（统计条右侧，可选）

```
已制作 144/156  ◔ 92%
```

- 全量热词中已制作比例  
- <100% 时显示「尚有 N 条待制作」提示链接，点击切到「待制作」Tab  

#### 5.2.6 卡片视图（Phase 4 可选）

表格为主；卡片视图适合词条少时：

### 5.3 ③ 存储空间监控

**目标**：防止 `cache/` 占满磁盘；热词越多，占用越直观。

#### 5.3.1 存储监控条（面板第一行，最醒目）

```
磁盘  ████████░░░░░░░░░░░░  48 MB / 500 MB  (10%)
      ↑ 进度条颜色随占用变化
```

| 占用率 | 进度条色 | 行为 |
|--------|----------|------|
| 0–70% | 绿 `#6ecf8a` | 正常 |
| 70–90% | 黄 `#f0b45c` | 统计条右侧 ⚠ 「空间偏紧」 |
| 90–100% | 红 `#e85d5d` | 弹窗建议清理 / 提高上限；**禁止**批量预合成直到处理 |
| ≥100% | 红 + 闪烁 | 仅允许播放已制作条目；新合成拒绝 |

#### 5.3.2 分项占用（点击「磁盘」展开 Tooltip 或折叠行）

| 分项 | 路径 | 示例 |
|------|------|------|
| 片段 WAV | `cache/moss/Junhao/*.wav` | 45 MB |
| 整句 L2 | `cache/utterance/*.wav` | 2 MB |
| 索引 | `index.json` + `hotwords.json` | <1 MB |
| **合计** | `cache/` | 48 MB |

#### 5.3.3 上限设置

`[调整上限]` → 小弹窗：

```
缓存容量上限 (MB)  [ 500  ]
条目数上限         [ 500  ]
□ 达到 90% 时自动 LRU 淘汰低频条目（保留固定词）
                    [保存]
```

写入 `config.local.json` → `cache.max_disk_mb` / `cache.max_entries`。

#### 5.3.4 单行「大小」列

- 已制作：显示该词条 WAV 实际字节（如 `42 KB`）  
- 待制作：`—`  
- 表尾可选汇总：`列表合计 45 MB`（仅已制作行求和）

#### 5.3.5 导入历史前的空间预估

步骤 2 预览弹窗必须显示：

> 预计新增 **87** 条，约 **6.5 MB**；当前剩余 **452 MB**，**可以导入**。

若超出剩余空间 → 按钮变灰，提示「请先清理或提高上限」。

### 5.4 ④ 手动添加热词

**目标**：用户主动添加机房核心设备名、常用短语，并可选立即 MOSS 制作。

#### 5.4.1 输入区布局（面板底部固定）

```
┌─ 手动添加热词 ────────────────────────────────────────────────────────┐
│  [ 输入词组 … ]  [添加为热词]  [添加并立即合成]  [试拆分整句]          │
│  提示: 请先经过符号读法转换；与热词表「词组」列一致方可命中              │
└───────────────────────────────────────────────────────────────────────┘
```

#### 5.4.2 交互规则

| 操作 | 行为 |
|------|------|
| **添加为热词** | 规范化 → 查重 → 写入 `hotwords.user_words`，状态 **待制作** |
| **添加并立即合成** | 同上 + 入队 `WarmupJob` 单条优先合成 |
| **试拆分整句** | 输入 `B05柜，NE20E杠1，端口，联通` → 弹窗展示 4 段及各自是否已制作 |
| 重复添加 | Toast「已在热词表中」，焦点定位到已有行 |
| 空 / 超长 | 校验：1–32 字，禁止纯符号 |

#### 5.4.3 批量粘贴（Phase 4）

多行文本框模式：每行一条，最多 50 行 → `[批量添加]`。

#### 5.4.4 与历史来源的优先级

同文本已存在时：

```
系统固定词 > 已制作 > 手动 > 历史
```

不重复占用条目上限；手动添加可 **提升** 历史条目的优先级（标记 `pinned: true`）。

### 5.5 ⑤ 特殊符号读法配置

**目标**：`-` 读「杠」等规则 **可配置**，且与缓存 Key、历史分析、手动添加 **共用同一套规范化**。

#### 5.5.1 入口

- 预缓存面板标题行：`[符号读法 ⚙]`  
- 或存储条右侧齿轮菜单 → 「符号读法设置」

#### 5.5.2 设置弹窗线框

```
┌─ 符号读法设置 ────────────────────────────────────────────────┐
│  说明: 保存后作用于 历史分析 / 手动添加 / 播报润色 / 缓存 Key   │
│  □ 启用符号读法（关则仅做基础空格清理）                        │
├─ 内置规则（可改读法，不可删符号）──────────────────────────────┤
│  符号    读法        示例输入 → 规范化结果          启用       │
│  -       [杠]       NE20E-1  →  NE20E杠1            [✓]      │
│  _       [下划线]   B05_NE20E → (设备名解析规则)     [✓]      │
│  /       [斜杠]     eth0/1   →  eth0斜杠1            [ ]      │
├─ 状态词（英文→中文，端口表）───────────────────────────────────┤
│  up      [联通]                                      [✓]      │
│  down    [中断]                                      [✓]      │
├─ 自定义规则 ──────────────────────────────────────────────────┤
│  #       [井号]                                      [✓]      │
│  [+ 添加规则]                                                 │
├─ 预览 ────────────────────────────────────────────────────────┤
│  输入: B05_NE20E-1 端口 down                                  │
│  输出: B05柜，NE20E杠1，端口，中断                             │
│                                    [恢复默认]  [取消]  [保存]  │
└───────────────────────────────────────────────────────────────┘
```

#### 5.5.3 与现有代码关系

当前 `alarm_processor.polish_speech_for_tts` 硬编码：

- `-` → `杠`（正则）  
- `up/down` → `联通/中断`  
- `B05_NE20E-1` → `B05柜，NE20E杠1`（`format_device_for_tts`）

**改造方向**（见拼合方案 §5.5）：

1. 规则迁入 `cache/symbol_rules.json` 或 `config.local.json`  
2. `normalize_for_cache(text)` = 符号规则 + polish  
3. **缓存 Key 必须用规范化后文本**，否则 `-` 与 `杠` 会存两份 WAV  

#### 5.5.4 修改规则后的缓存策略

| 选项 | 行为 |
|------|------|
| 仅影响新条目（默认） | 旧 WAV 保留，Key 不变 |
| 规则变更较大时提示 | 「是否标记全部待制作条目需重新合成？」 |

## 七、与侧栏 / 顶栏的联动

### 7.1 引擎切换

| 语音引擎 | 右下预缓存面板 | 顶栏缓存徽章 | 左栏拼合开关 |
|----------|----------------|--------------|--------------|
| MOSS 本地 | **显示** | 显示 | 显示 |
| Windows SAPI | **隐藏**（或折叠占位提示） | 隐藏 | 隐藏 |
| 讯飞在线 | 隐藏 | 隐藏 | 隐藏 |

### 7.2 模型预热 vs 片段预缓存

```
┌─────────────────────────────────────────────────────────┐
│ 左栏：启动时预热模型（勾选）                              │
│   → moss_daemon 加载 ONNX 到内存（30–90s，一次性）        │
│   → 没有它，片段 MOSS 合成无法进行                        │
├─────────────────────────────────────────────────────────┤
│ 右下：MOSS 语音预缓存                                     │
│   → 把文本片段合成 WAV 存 disk（批量，可后台）            │
│   → 播报时拼合 WAV，不再跑神经网络                        │
└─────────────────────────────────────────────────────────┘
```

**推荐引导文案**（首次选 MOSS 时在预缓存面板顶显示一次）：

> ① 先在左侧勾选「启动时预热模型」→ ② 再点「预热固定词」→ ③ 试听同一句，应明显变快。

### 7.3 顶栏缓存徽章

- 文案：`缓存 82%` 或 `156 条`  
- 点击：**展开**右下预缓存面板（若已折叠则展开并 scroll_into_view）  
- 颜色：命中率 ≥70% 绿底，否则黄底  

### 7.4 数据终端与预缓存的信息分工

| 信息 | 数据终端 | 预缓存面板 |
|------|----------|------------|
| 告警事件原文 | 是 | 否 |
| 单次合成摘要 `拼合 4/4 · 280ms` | 是（每条事件下缩进） | **最近一条**摘要 D6 |
| 热词列表 | 否 | 是 |
| 批量预热 37/250 | 可选一行 | **进度条主展示** |
| MISS 正在合成哪一段 | 是（实时） | 热词表对应行 ⟳ |

终端摘要格式（与 v5 §6.2 一致）：

```
[14:25:25] 试听 · MOSS 本地
  文本: B05柜，NE20E杠1，端口，联通
  合成: 拼合 4/4 命中 | 280ms | 静音 80ms
```

## 九、核心交互流程（五大功能）

### 流程 A — 历史预生成（①）

1. 点 **从历史生成热词** → 选 500 条 → **分析**。  
2. 预览 87 条新增 + 预计 6.5 MB → **确认并导入**。  
3. 勾选 **自动预合成** → 进度条 37/250 → 热词表行变 **已制作**。  
4. 存储条更新：`48 MB → 54 MB`，制作进度环 92% → 100%。  

### 流程 B — 观察热词表（②）

1. 打开 **待制作** Tab → 见 12 条灰色行。  
2. 点 **预合成全部待制作** 或单行 **合成**。  
3. 行变绿 **已制作**，**大小** 列出现 KB 数。  

### 流程 C — 存储将满（③）

1. 磁盘条变黄 `450/500 MB (90%)`。  
2. 弹窗：启用 LRU 或 **清空低频** / **提高上限**。  
3. 历史导入前若预估超限 → **确认** 按钮禁用。  

### 流程 D — 手动热词（④）

1. 输入 `NE20E杠2` → **添加并立即合成**。  
2. 新行来源 **手动**，状态 **制作中** → **已制作**。  
3. 试听验证 MOSS 音质。  

### 流程 E — 符号读法（⑤）

1. 点 **符号读法 ⚙** → 把 `/` 读法改为 **斜杠** 并启用。  
2. **预览** 区确认 `eth0/1` → `eth0斜杠1`。  
3. **保存** → 此后手动添加、历史分析均用新规则。

## 十二、验收标准（v6.2）

### 缓存五大功能（v6.1）

1. **①** 可从历史 N 条分析热词，预览条数/体积，并后台预合成。  
2. **②** 热词表区分 **已制作/待制作/制作中/失败**，来源 **系统/历史/手动** 一目了然。  
3. **③** 存储条显示 **已用/上限 MB + 进度条**，≥90% 有警告；导入前有空闲校验。  
4. **④** 可手动添加热词，支持 **添加并立即合成**。  
5. **⑤** 符号读法可配置，预览生效，且与缓存 Key 规范化一致。  

### 数据终端滚动（v6.2 新增）

6. 预缓存展开、终端高度约 55% 时，日志超过可视区 → **右侧垂直滚动条**可用。  
7. 滚轮 / 触控板 / 方向键可在终端内上下浏览，**不撑高**右栏外框。  
8. 新日志到达：用户未上滚则 **自动滚到底**；手动上滚后可用 **[回到底部]** 恢复。  
9. 折叠预缓存后终端变高，滚动能力仍保留。  
10. 五项缓存功能均在终端下方同一面板；非 MOSS 引擎时隐藏预缓存区。
